Theorie: POSIX‑ACL Syntax & Mask

  • Form: u|g:[name]:[rwx], m:[rwx] (mask), o:[rwx], sowie d: für Default auf Verzeichnissen.
  • Mask: Kappung der effektiven Rechte für alle Gruppen/Named‑User‑Einträge (außer user::).
  • Effektive Rechte werden in getfacl mit #effective: angezeigt.

Advanced: setfacl/getfacl in der Praxis

# 1) Einzelne Einträge setzen
setfacl -m u:alice:rw file
setfacl -m g:dev:rwx file

# 2) Maske explizit setzen (sonst kappen!)
setfacl -m m:rwx file

# 3) Default-ACLs auf Verzeichnissen (Vererbung)
setfacl -m d:g:dev:rwx dir

# 4) Rekursiv anwenden (Vorsicht; prüfen!)
setfacl -R -m g:dev:rx dir
getfacl -R dir | less

# 5) Entfernen/Löschen
setfacl -x u:alice file
setfacl -b file  # alle ACLs löschen

# 6) Kopieren/Restore
getfacl -R /srv/proj > /root/proj.acl
setfacl --restore=/root/proj.acl

# 7) Nur Default-ACLs exportieren (Filter)
getfacl -R /srv/proj | awk '/^default:/'

Integration: rsync/tar und Netzwerkfreigaben

  • rsync: Flag -A für ACLs, -X für xattrs, häufig kombiniert mit -aAX.
  • tar: GNU tar mit --acls --xattrs nutzen.
  • Samba/NFS: Server‑ und Export‑Optionen müssen ACLs unterstützen; ggf. Mapping beachten.
rsync -aAX --delete /src/ /dst/
tar --acls --xattrs -cpf backup.tar /srv/proj

Sicherheit & Audits

  • Inventar: getfacl -R regelmäßig sichern, Änderungen versionieren.
  • Review: Vermeide viele user‑spezifische ACLs; bevorzuge Gruppen‑basierte Einträge.
  • Suche nach Ausreißern: Welt‑schreibbare Dateien, ungewollte Ausführungsrechte finden.
find /srv/proj -type f -perm -o+w -ls
find /srv/proj -type d -perm -o+w -ls

Troubleshooting

  • Mask‑Effekt: Erwartete rwx wird zu r-x → setfacl -m m:rwx.
  • Kopie ohne ACLs: Nutze rsync -A oder cp --preserve=all (je nach System).
  • Default‑ACL greift nicht: Nur Verzeichnisse haben Default‑ACL; Objekt wurde vorher erstellt?

Vertiefung & Praxis

Vertiefung: Dateirechte, Sonderbits und ACLs.

  • Symbolische vs. oktale Modi (u+rw,g-w,o= vs. 640).
  • Sonderbits: suid (u+s), sgid (g+s), sticky (o+t).
  • ACLs erlauben benutzerspezifische Rechte zusätzlich zu U/G/O.
# Standardrechte via umask (Beispiel: 022)
umask 022

# Klassische Rechte
chmod 640 datei
chmod u+rwx,g+rx,o-rwx script.sh
chown alice:proj datei

# Sonderbits
chmod u+s /usr/local/bin/tool
chmod g+s /srv/proj
chmod o+t /tmp

# ACLs
setfacl -m u:alice:rw- -m g:proj:r-- datei
getfacl datei

# Default-ACLs für neue Dateien in Ordnern
setfacl -d -m g:proj:rwX /srv/proj

Fehlerbilder

  • ACL mask reduziert effektive Rechte → getfacl prüfen (mask).
  • chmod 777 als „Fix“ → in der Prüfung tabu; gezielt Rechte setzen.

Prüfungstipps

  • Rechenweg: umask bestimmt abgezogene Bits (Basis 666/777).
  • X in ACL (großes X) setzt x nur, wenn es ein Ordner ist oder x bereits irgendwo gesetzt war.

Checkliste – schneller Durchblick

  • Manpage/--help gelesen und Option bewusst gewählt?
  • Rechte/Ownership/SELinux/AppArmor berücksichtigt?
  • Distribution & Version: gleiches Verhalten verifiziert?

Optionen‑Atlas (Kernbefehle)

KommandoWichtige Optionen

Szenarien aus der Praxis

  • Gemeinsamer Teamordner mit Default‑ACL für Gruppe proj (rwX).
  • Sticky-Bit auf geteilten Ordnern (z. B. /tmp, /shared) gegen versehentliches Löschen.
  • Setuid‑Hilfsprogramm: Sicherheitsprüfung und Alternativen (sudo mit NOPASSWD).
  • Backup-User erhält Leserechte per ACL ohne Gruppenwechsel.

Diagnose-Flow (schneller Pfad zur Lösung)

  1. Ownership korrekt? (ls -l)
  2. U/G/O‑Rechte passend? (chmod)
  3. ACLs vorhanden & Mask? (getfacl)
  4. Sonderbits sinnvoll? (suid/sgid/sticky)
