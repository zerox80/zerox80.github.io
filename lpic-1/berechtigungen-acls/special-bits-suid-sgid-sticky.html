Übungen

  1. Warum hat /usr/bin/passwd suid?

Prüfungsfragen

  1. Wie erkennst du das Sticky‑Bit in ls -l?

Theorie

  • setuid (u+s): Prozesse erben Effektiv‑UID des Dateieigentümers. Beispiel: passwd (root‑owned) darf /etc/shadow ändern.
  • setgid (g+s): Analog für Gruppen. Auf Verzeichnissen sorgt g+s, dass neue Dateien die Gruppen‑ID des Verzeichnisses erben.
  • sticky (t): Auf Verzeichnissen: Nur Eigentümer (oder root) darf Einträge löschen/umbenennen. Typisch: /tmp.
  • Darstellung in ls -l: -rwsr-xr-x (s statt x für suid), -rwxr-sr-x (s für sgid), drwxrwxrwt (t für sticky).

Sicherheitsaspekte

  • Angriffsfläche minimieren: SUID/SGID‑Binaries nur wenn absolut nötig. Entferne auf lokalen Tools suid, falls unnötig.
  • Pfad‑Hijacking vermeiden: SUID‑Programme sollten harte Pfade nutzen und PATH/IFS/LD_* Variablen absichern.
  • Dateirechte & Eigentümer prüfen: SUID‑Dateien müssen root‑owned und unveränderbar durch Unprivilegierte sein.
  • Mount‑Optionen: Auf gemeinsam genutzten Medien nodev,nosuid,noexec in fstab nutzen.
# Audit: suid/sgid finden
find / -xdev \( -perm -4000 -o -perm -2000 \) -type f -ls 2>/dev/null

# Härtung: suid entfernen (Beispiel, mit Vorsicht!)
chmod u-s /usr/local/bin/tool

Praxis‑Patterns

  • Teamverzeichnisse: chmod g+s /srv/team plus passende Gruppe → konsistente Gruppenbesitzrechte.
  • Upload‑Ordner: chmod 1777 /srv/upload (sticky) verhindert Fremdlöschung.
  • Build‑Artefakte: Kein suid/sgid auf Artefakten; CI/CD prüft Rechte vor Release.
# Teamordner
install -d -m 2775 -o root -g dev /srv/team   # g+s

# Upload wie /tmp
install -d -m 1777 /srv/upload                 # sticky

Troubleshooting

  • Sticky nicht wirksam: Prüfe, ob das Verzeichnis wirklich +t trägt und ob der löschende Nutzer Owner ist.
  • Gruppenerbe fehlt: g+s auf Elternverzeichnis vergessen. chmod 2775 DIR.
  • SUID ignoriert: Auf dem Dateisystem/Mount gilt nosuid?

Erweiterte Übungen

  1. Richte einen Teamordner mit g+s und korrekter Gruppenzugehörigkeit ein. Verifiziere mit zwei Testnutzern.
  2. Erzeuge einen Upload‑Ordner mit sticky. Teste Löschverhalten zwischen unterschiedlichen Besitzern.
  3. Inventarisiere SUID/SGID im System und dokumentiere, welche benötigt werden.
Lösungshinweise
groupadd team; usermod -aG team alice; usermod -aG team bob

Vertiefung & Praxis

Vertiefung: Dateirechte, Sonderbits und ACLs.

  • Symbolische vs. oktale Modi (u+rw,g-w,o= vs. 640).
  • Sonderbits: suid (u+s), sgid (g+s), sticky (o+t).
  • ACLs erlauben benutzerspezifische Rechte zusätzlich zu U/G/O.
# Standardrechte via umask (Beispiel: 022)
umask 022

# Klassische Rechte
chmod 640 datei
chmod u+rwx,g+rx,o-rwx script.sh
chown alice:proj datei

# Sonderbits
chmod u+s /usr/local/bin/tool
chmod g+s /srv/proj
chmod o+t /tmp

# ACLs
setfacl -m u:alice:rw- -m g:proj:r-- datei
getfacl datei

# Default-ACLs für neue Dateien in Ordnern
setfacl -d -m g:proj:rwX /srv/proj

Fehlerbilder

  • ACL mask reduziert effektive Rechte → getfacl prüfen (mask).
  • chmod 777 als „Fix“ → in der Prüfung tabu; gezielt Rechte setzen.

Prüfungstipps

  • Rechenweg: umask bestimmt abgezogene Bits (Basis 666/777).
  • X in ACL (großes X) setzt x nur, wenn es ein Ordner ist oder x bereits irgendwo gesetzt war.

Checkliste – schneller Durchblick

  • Relevante Logdateien geprüft? (journalctl, /var/log/*)
  • Manpage/--help gelesen und Option bewusst gewählt?
  • Rechte/Ownership/SELinux/AppArmor berücksichtigt?
  • Distribution & Version: gleiches Verhalten verifiziert?

Szenarien aus der Praxis

  • Gemeinsamer Teamordner mit Default‑ACL für Gruppe proj (rwX).
  • Sticky-Bit auf geteilten Ordnern (z. B. /tmp, /shared) gegen versehentliches Löschen.
  • Setuid‑Hilfsprogramm: Sicherheitsprüfung und Alternativen (sudo mit NOPASSWD).
  • Backup-User erhält Leserechte per ACL ohne Gruppenwechsel.

Diagnose-Flow (schneller Pfad zur Lösung)

  1. Ownership korrekt? (ls -l)
  2. U/G/O‑Rechte passend? (chmod)
  3. ACLs vorhanden & Mask? (getfacl)
  4. Sonderbits sinnvoll? (suid/sgid/sticky)
