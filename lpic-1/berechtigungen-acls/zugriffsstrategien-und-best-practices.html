Zugriffsstrategien & Best Practices

Unterthemen

Rechte‑Grundlagen chmod, chown, chgrp umask & Defaults Special Bits (suid/sgid/sticky) ACL‑Grundlagen ACL‑Werkzeuge Best Practices

Lernziele

  • Sichere Standardrechte definieren.
  • Teams und Projekte sinnvoll strukturieren.

Cheatsheet

# Projektstruktur
mkdir -p /srv/proj/{data,bin}
chgrp -R proj /srv/proj
chmod 2775 /srv/proj/data   # g+s
setfacl -m d:g:proj:rwx /srv/proj/data

Übungen

  1. Lege ein Projektverzeichnis mit Gruppenzugehörigkeit an.

Prüfungsfragen

  1. Wann ist g+s auf Verzeichnissen sinnvoll?

Theorie & Strategien

  • Least Privilege: Nur notwendige Rechte gewähren (Datei‑, Verzeichnis‑, Ausführungsrechte).
  • Gruppenbasiert statt benutzerbasiert: Rollen über Gruppen abbilden; vermeidet Einzel‑Sonderfälle.
  • Trennung von Daten & Code: Schreibrechte für App‑Daten, aber nur Lese‑/Ausführungsrechte für Deploy‑Artefakte.
  • Standardrechte steuern: umask und Default‑ACLs definieren (z. B. Team‑Shares).
  • Vererbung via g+s & Default‑ACLs: Konsistente Gruppenzugehörigkeit und Startberechtigungen in Teamordnern.

Rollen & Gruppenmodell

Beispielhafte Rollen: owner, maintainer, developer, viewer. Diese werden Gruppen zugeordnet und über ACLs feinjustiert.

# Gruppen anlegen
groupadd proj-admins
groupadd proj-dev
groupadd proj-read

# Projektbasis
install -d -m 2775 -o root -g proj-admins /srv/proj
install -d -m 2775 -o root -g proj-dev    /srv/proj/data
install -d -m 2775 -o root -g proj-read   /srv/proj/docs

# ACL-Default für Vererbung
setfacl -R -m d:g:proj-admins:rwx /srv/proj
setfacl -R -m d:g:proj-dev:rwx    /srv/proj/data
setfacl -R -m d:g:proj-read:rx    /srv/proj/docs

Umask & Defaultrechte Policy

  • Service‑Kontexte: systemd‑Services mit restriktiver UMask=0077 in Unit‑Dateien.
  • User‑Shells: Standard umask 002 in kollaborativen Teams (Gruppenschreibrechte), ansonsten 022.
  • Team‑Shares: Default‑ACLs nutzen statt nur umask, um konsistente Gruppenrechte zu erzwingen.
# systemd Override
systemctl edit myapp.service
[Service]
UMask=0077

ACL‑Patterns (POSIX)

  • Team‑Ordner: Default‑ACLs für Gruppen schalten Vererbung ein.
  • Externe Zugriffe: Temporäre, enge ACLs für Audits oder Support; Ablaufdatum dokumentieren.
  • Mask beachten: Die mask begrenzt die effektiven Gruppen‑/Benutzer‑ACLs.
# Team-Ordner mit Default-ACLs
setfacl -m g:proj-dev:rwx /srv/proj/data
setfacl -m d:g:proj-dev:rwx /srv/proj/data

# Leserolle
setfacl -m g:proj-read:rx  /srv/proj/docs
setfacl -m d:g:proj-read:rx /srv/proj/docs

# Effektive Rechte anzeigen
getfacl /srv/proj/data

Special Bits sinnvoll einsetzen

  • setgid auf Verzeichnissen (g+s): Neue Dateien erben die Gruppen‑ID → konsistente Team‑Besitzverhältnisse.
  • sticky bit auf gemeinsam genutzten Ordnern: /tmp‑ähnliches Verhalten: Nur Eigentümer darf löschen.
  • SUID/SGID BINARIES: Streng minimieren, regelmäßig auditieren (find / -perm -4000 -o -perm -2000).

Service‑ & Systemkonten

  • Dedizierte Benutzer/Gruppen pro Dienst, keine geteilten Konten.
  • Datei‑ und Verzeichnisrechte für Runtime‑Daten separieren (/var/lib/app, /var/log/app).
  • Mit ReadOnlyPaths=, ReadWritePaths=, ProtectSystem=strict in systemd weiter einschränken.
# systemd Härtung (Ausschnitt)
[Service]
ProtectSystem=strict
ReadWritePaths=/var/lib/myapp /var/log/myapp

Auditing & Nachvollziehbarkeit

  • Rechteinventar: Regelmäßig getfacl -R sichern; Änderungen per Versionskontrolle tracken.
  • Auditd/Journal: Kritische Pfade überwachen (auditctl -w /srv/proj -p wa -k proj).
  • Finde Abweichungen: find für zu weite Rechte, SUID/SGID, Welt‑schreibbare Dateien.
# Backups der ACLs
getfacl -R /srv/proj > /root/acl-backup.srv-proj

# Suche potenziell gefährliche Rechte
find /srv/proj -type f -perm -o+w -ls
find / -xdev \( -perm -4000 -o -perm -2000 \) -ls

Troubleshooting

  • ACL mask greift: Effektivrechte kleiner als erwartet → setfacl -m m:rwx anpassen (vorsichtig, dokumentieren).
  • Falsche Gruppenerben: chmod g+s auf Eltern‑Verzeichnis vergessen.
  • Umask überschreibt Erwartungen: Shell/Service‑Kontext prüfen, ggf. Default‑ACLs nutzen.

Erweiterte Übungen

  1. Implementiere ein dreistufiges Rollenmodell (admin/dev/read) für /srv/projX inkl. Default‑ACLs.
  2. Härten einen gemeinsam genutzten Upload‑Ordner mit sticky, g+s und restriktiven Default‑ACLs.
  3. Richte ein Audit auf /srv/projX/data ein und belege Änderungen mit Logs.
Lösungshinweise
# Rollen
groupadd px-admins; groupadd px-dev; groupadd px-read
install -d -m 2775 -o root -g px-admins /srv/projX
install -d -m 2775 -o root -g px-dev    /srv/projX/data
setfacl -m g:px-admins:rwx /srv/projX
setfacl -m d:g:px-admins:rwx /srv/projX
setfacl -m g:px-dev:rwx /srv/projX/data
setfacl -m d:g:px-dev:rwx /srv/projX/data
setfacl -m g:px-read:rx /srv/projX
setfacl -m d:g:px-read:rx /srv/projX

# Upload-Ordner
install -d -m 1770 -o root -g px-dev /srv/projX/upload
setfacl -m d:g:px-dev:rwx /srv/projX/upload

Checkliste (Policy)

  • Gruppen vor Benutzern? Rollenkonzept dokumentiert.
  • g+s & Default‑ACLs? Teamordner korrekt vererbend konfiguriert.
  • Umask/Service‑UMask? Kontextspezifisch gesetzt und geprüft.
  • Audits vorhanden? ACL‑Backups, auditd‑Regeln, regelmäßige Prüfungen.
  • SUID/SGID geprüft? Minimiert und inventarisiert.

Szenarien aus der Praxis

  • Gemeinsamer Teamordner mit Default‑ACL für Gruppe proj (rwX).
  • Sticky-Bit auf geteilten Ordnern (z. B. /tmp, /shared) gegen versehentliches Löschen.
  • Setuid‑Hilfsprogramm: Sicherheitsprüfung und Alternativen (sudo mit NOPASSWD).
  • Backup-User erhält Leserechte per ACL ohne Gruppenwechsel.

Diagnose-Flow (schneller Pfad zur Lösung)

  1. Ownership korrekt? (ls -l)
  2. U/G/O‑Rechte passend? (chmod)
  3. ACLs vorhanden & Mask? (getfacl)
  4. Sonderbits sinnvoll? (suid/sgid/sticky)
