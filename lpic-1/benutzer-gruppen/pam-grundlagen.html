Logins laufen durch die entsprechenden Stacks (auth, account, password, session). Reihenfolge und Flags bestimmen das Verhalten.

Theorie & Konzepte

Stacks & Control‑Flags

Jeder Dienst besitzt Stacks für auth, account, password, session. Wichtige Flags:

  • required: Muss bestehen; Fehler werden gesammelt, Entscheidung am Ende negativ.
  • requisite: Muss bestehen; bei Fehler sofortiger Abbruch.
  • sufficient: Bei Erfolg ggf. Kurzschluss, wenn zuvor kein required scheiterte.
  • optional: Zählt nur, wenn alleinstehend.

Praxis: Typische Konfigurationen

# /etc/pam.d/sudo (Beispiel)
auth       required   pam_env.so
auth       required   pam_unix.so
account    required   pam_unix.so
password   required   pam_unix.so
session    required   pam_limits.so

# /etc/pam.d/sshd (Ausschnitte; Distro-abhängig)
auth       required   pam_env.so
auth       sufficient pam_unix.so try_first_pass
auth       [default=die] pam_faillock.so authfail
auth       sufficient pam_faillock.so authsucc

# /etc/pam.d/login (Ausschnitte)
auth       required   pam_securetty.so
auth       requisite  pam_nologin.so
auth       required   pam_unix.so
session    required   pam_limits.so

Hinweis: Distro‑Defaults variieren. Vor Änderungen Backups in /etc/pam.d/ erstellen und in separatem Terminal testen.

Account‑Sperren mit pam_faillock

# Häufig in system-auth/common-auth inkludiert
auth   required   pam_faillock.so preauth silent audit deny=5 unlock_time=600
auth   [success=1 default=bad] pam_unix.so
auth   [default=die] pam_faillock.so authfail audit
account required   pam_faillock.so

# Status prüfen / zurücksetzen
faillock --user alice
faillock --user alice --reset

Limits & Umgebungsvariablen

pam_limits liest /etc/security/limits.conf sowie limits.d/ (z. B. nofile, nproc). pam_env setzt Variablen aus /etc/environment bzw. environment.d/.

Best Practices

  • Backup & Test: Vor Änderungen Dateien sichern; neue Sessions in zweitem Terminal testen, um Lockouts zu vermeiden.
  • Reihenfolge: Flags wie sufficient können spätere Checks überspringen – bewusst einsetzen.
  • Logging: Module mit debug/audit nutzen und journalctl prüfen.

Cheatsheet

# Orte
/etc/pam.d/           # Service-Dateien

# Flags (Kurz)
required  = Fehler am Ende
requisite = Fehler bricht sofort ab
sufficient= Erfolg kann kurzschließen
optional  = zählt nur allein

# Faillock
faillock --user NAME          # Status
faillock --user NAME --reset  # Reset

# Limits-Beispiel (/etc/security/limits.d/90-custom.conf)
*  soft  nofile  65535
*  hard  nofile  65535

Übungen

  1. Fügen Sie eine Anmeldebegrenzung mit pam_faillock (5 Fehlversuche, 10 Minuten) hinzu. Demonstrieren Sie Lock/Unlock.
  2. Konfigurieren Sie pam_limits für erhöhte nofile-Werte und verifizieren Sie diese mit einer neuen Login‑Session.
  3. Erklären Sie den Unterschied zwischen required und requisite anhand eines minimalen Stacks.

Zu meinem Fortschritt

Vertiefung & Praxis

Vertiefung: Benutzer- und Gruppenverwaltung.

  • Benutzer-Lebenszyklus: anlegen → Passwort setzen → Gruppenmitgliedschaften → Shell/Home → Ablaufdatum → Sperren/Entsperren.
  • Dateirechte & Ownership im Zusammenspiel mit primärer und sekundären Gruppen.
  • PAM-Module (Pluggable Authentication Modules): Reihenfolge und Kontrollflags (required, requisite, sufficient, optional).

Kommandoreferenz

# Benutzer anlegen (mit Home, Standardgruppe, Shell)
sudo useradd -m -s /bin/bash -U alice
sudo passwd alice

# Benutzer ändern (Shell, Gruppen, Ablauf)
sudo usermod -s /bin/zsh -aG wheel,audio alice
sudo chage -E 2025-12-31 alice

# Benutzer sperren/entsperren
sudo usermod -L alice   # sperren
sudo usermod -U alice   # entsperren

# Benutzer löschen (+Home)
sudo userdel -r alice

# Gruppenverwaltung
getent group developers
sudo groupadd developers
sudo gpasswd -a alice developers
sudo gpasswd -d alice developers

Praxis-Lab

  1. Lege proj‑Gruppe an, füge 2 Nutzer hinzu.
  2. Erstelle /srv/proj und setze sgid, damit neue Dateien Gruppen‑Ownership erben.
  3. Erzwinge sichere Passwörter via pam_pwquality.so (minlen, ucredit/lcredit/dcredit).

Häufige Fehler

  • Permission denied trotz Mitgliedschaft → neu einloggen (Gruppen gelten erst für neue Sessions).
  • nologin-Shell gesetzt → interaktive Anmeldung scheitert.
  • Home nicht erstellt: useradd -m vergessen.

Prüfungstipps

  • Unterschied primäre vs. sekundäre Gruppe kennen.
  • chage‑Felder: minimale/maximale Tage, Warnzeit, Inaktivität.
  • visudo verwenden, niemals /etc/sudoers direkt editieren.

Checkliste – schneller Durchblick

  • Relevante Logdateien geprüft? (journalctl, /var/log/*)
  • Manpage/--help gelesen und Option bewusst gewählt?
  • Rechte/Ownership/SELinux/AppArmor berücksichtigt?
  • Distribution & Version: gleiches Verhalten verifiziert?

Szenarien aus der Praxis

  • Projektordner /srv/proj mit sgid: neue Dateien erben Gruppen‑Ownership.
  • Service-Account ohne Login: Shell /usr/sbin/nologin und keine Home-Erstellung.
  • Passwortpolitik mit pam_pwquality: Mindestlänge & Zeichencocktail erzwingen.
  • Temporäre Adminrechte via sudoers Drop-In in /etc/sudoers.d/NAME.

Diagnose-Flow (schneller Pfad zur Lösung)

  1. Existiert der Nutzer/Gruppe? (getent passwd/group)
  2. Richtige Shell/Home gesetzt? (useradd/usermod -s/-m)
  3. Passwort/Lock‑Status ok? (passwd -S, chage -l)
  4. Gruppenmitgliedschaften aktiv? (id; neue Session nötig)
