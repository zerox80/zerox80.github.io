FHS – Filesystem Hierarchy Standard

Zusammenfassung

Der Filesystem Hierarchy Standard (FHS) definiert Konventionen für die Verzeichnisstruktur unter Linux. Dadurch bleiben Systeme vorhersehbar und Scripts portabel.

PfadBeschreibung
/Wurzel des Verzeichnisbaums
/etcSystemweite Konfiguration
/usrSystemprogramme, Bibliotheken, Dokumentation
/varVariable Daten (Logs, Spools)
/homeBenutzerverzeichnisse
/bin, /sbinEssentielle Binaries für System/Boot (oft nach /usr/bin verlinkt)
/lib, /lib64Essentielle Bibliotheken (teils nach /usr/lib* verlinkt)
/bootKernel, Initramfs, Bootloader‑Dateien
/devGerätedateien (udev generiert dynamisch)
/proc, /sysPseudofs mit Kernel/Hardware‑Infos
/runflüchtige Laufzeitdaten (tmpfs)
/tmpTemporäre Dateien (i.d.R. beim Booten geleert)
/srvDaten von Diensten (z. B. Web, FTP)
/optOptionale Zusatzsoftware (vendor‑spezifisch)
/media, /mntWechselmedien (automount), temporäre Mounts
/rootHome‑Verzeichnis von root

Prüfungsfragen

  1. Was ist der Zweck von /usr vs. /bin?

Theorie: FHS im Detail

  • /usr: Meist nur lesbar; enthält /usr/bin, /usr/sbin, /usr/lib*, /usr/share. Distribution‑Pakete installieren hier.
  • /etc: Konfiguration im Textformat; keine großen Daten, keine Binaries. Beispiele: /etc/fstab, /etc/ssh/sshd_config.
  • /var: Schreibintensiv: /var/log, /var/spool, /var/lib (persistente App‑Daten), /var/cache.
  • /tmp vs. /var/tmp: /tmp kurzlebig (oft beim Booten geleert), /var/tmp längerlebig.
  • /opt: Dritthersteller‑Software, Versionierung möglich: /opt/VENDOR/produkt/.
  • /srv: Daten, die von Diensten bereitgestellt werden, z. B. /srv/www.
  • /proc: Prozess/Kerneldaten; z. B. /proc/cpuinfo, /proc/meminfo, /proc/sys (sysctl).
  • /sys: sysfs (Geräte, Treiber, Klassen) – udev/hotplug.
  • /run: Laufzeit‑State (PIDs, Sockets); ersetzt alte Pfade in /var/run.

Praxis

# Überblick gewinnen (Top‑Level)
tree -L 1 -a /

# Zuordnen: welche Pakete liefern Dateien?
dpkg -S /usr/bin/ls     # Debian/Ubuntu
rpm -qf /usr/bin/ls     # RHEL/SUSE

# Logs durchsuchen
journalctl -b
sudo tail -F /var/log/syslog   # Debian/Ubuntu
sudo tail -F /var/log/messages # RHEL/SUSE

# Temporäres Verzeichnis korrekt
stat -c "%a %n" /tmp  # erwartet 1777 (sticky)

# Systemweite Standards
grep -R "^#\|^[^#]" -n /etc/default/ /etc/sysconfig/ 2>/dev/null | head

# Ort für Dienstdaten vorbereiten
sudo install -d -o root -g root -m 0755 /srv/www

Best Practices

  • Trennung Lesen/Schreiben: /usr read‑only möglich; App‑Daten in /var/lib, Service‑Daten in /srv.
  • Rechte & Sticky Bits: /tmp und /var/tmp auf 1777; keine Welt‑schreibbaren Verzeichnisse ohne Sticky Bit.
  • Konfiguration versionieren: /etc (z. B. git‑Repo) mit Secrets‑Ausschlüssen.
  • Wechselmedien: unter /media (automount) statt wilde Pfade.
  • Vendor Software: in /opt sauber halten, keine Vermischung mit /usr.

Fehlersuche

  • Vollgelaufenes /var: du -xhd1 /var, Logrotation prüfen (logrotate), Caches leeren (apt clean, dnf clean all).
  • Falsche Rechte auf /tmp: sudo chmod 1777 /tmp, sudo chown root:root /tmp.
  • Read‑only Root: Mountstatus mit findmnt /, dmesg prüfen; ggf. Notfall‑Remount mount -o remount,rw / (nur bewusst!).
  • Verwaiste PIDs/Sockets in /run: betroffenen Dienst neu starten; systemd pflegt /run.
